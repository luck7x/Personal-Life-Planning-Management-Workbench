# 监督反思 MVP Web 任务工作台设计稿

## 1. 目标

本轮设计目标从“继续优化日历”切换为“先把 Web 端任务工作台做对”。

核心要解决的问题：

1. 当前任务页五列平铺，主次不分，整体布局怪异。
2. 空列占地过大，页面大量视觉面积被空白白卡消耗。
3. 任务卡信息和操作混在一起，卡片虚胖，任务密度低。
4. `Done` 长期堆积在主工作台，缺少自动沉淀机制。
5. 缺少一套清晰的 agent team 协作与前端执行约束。

本稿只定义 **Web 端任务工作台与归档模块**，不展开移动端细节，不继续优化日历视图。

## 2. 范围

### 2.1 本次纳入

- Web 端任务工作台信息架构重构
- Web 端任务卡片层级与详情交互重构
- Web 端垃圾篓入口重构
- `Done` 的每日自动归档机制
- 归档页信息架构与交互设计
- agent team 分工与前端执行规范

### 2.2 明确不做

- 移动端布局与交互实现
- 日历视图继续打磨
- 统计报表页
- 后端接入与云端同步
- 权限、多用户、协作能力

## 3. 设计原则

### 3.1 主次分明

主工作流只强调 `Todo -> Doing -> Done`。

`备忘 / 逾期 / 垃圾篓 / 归档` 都属于辅助态、异常态或历史态，不与主流程平权展示。

### 3.2 在线态与历史态分离

- 工作台承接“当前在线流转”
- 归档承接“过去已完成任务”

`Done` 不应长期堆在主工作台内。

### 3.3 信息与操作分层

- 任务卡默认负责“识别”
- 详情抽屉负责“查看、编辑、流转、删除”

卡面不常驻铺满操作按钮。

### 3.4 页面骨架稳定

- 页面整体不做无限长滚动
- 列内滚动承接内容溢出
- 用户优先看到最新流入和最新变更任务

## 4. Web 端信息架构

Web 端采用“两栏式工作台”：

- 左侧辅助栏
- 右侧主工作区
- 右侧详情抽屉（按需出现）

### 4.1 左侧辅助栏

左侧辅助栏只承载：

- `备忘`
- `逾期`

`垃圾篓` 不再占用完整区块，改为 **小图标按钮入口**。

建议结构：

1. 辅助栏顶部：标题、垃圾篓图标按钮
2. 中部：`备忘`
3. 下部：`逾期`

### 4.2 右侧主工作区

右侧主工作区只承载：

- `Todo`
- `Doing`
- `Done`

主区是最高优先级视觉区域，用于回答三个问题：

1. 现在待处理什么
2. 现在正在做什么
3. 今天刚完成了什么

### 4.3 次级入口

Web 顶部导航或工具位保留以下入口：

- `任务工作台`
- `归档`
- `日历`

其中本轮只细化前两者。

## 5. 页面骨架与留白策略

### 5.1 页面骨架

- 左侧辅助栏固定宽度：建议 `280px - 320px`
- 右侧主工作区：占满剩余空间
- 主区三列：等宽或近等宽分配

### 5.2 顶部区域

顶部不再保留大面积 hero 说明卡。

建议收敛为：

- 左侧：页面标题 + 一句简短说明
- 右侧：少量工具按钮与入口

任务工作台是操作页，不是展示页，首屏应优先露出任务而非文案。

### 5.3 留白策略

新布局保留必要呼吸感，但不再通过“巨大空列 + 巨大卡片”制造存在感。

留白规则：

- 页面外边距适中
- 列间距稳定
- 内容区优先，空状态降噪

### 5.4 空状态

空状态不再使用大面积白卡占位。

建议改为轻量占位：

- 虚线框
- 低饱和提示
- 可选快捷入口

示例文案：

- `暂无待办`
- `今天没有逾期任务`
- `拖入任务或新建`

## 6. 卡片体系

Web 端任务卡分为两级。

### 6.1 主区卡片

适用范围：

- `Todo`
- `Doing`
- `Done`

默认展示：

- 标题
- 分类
- 时间信息

不默认展示：

- 大量操作按钮
- 全量编辑字段

视觉要求：

- 白底卡片
- 左侧状态/分类色条
- 高度紧凑
- 一屏能容纳更多卡片

### 6.2 侧栏卡片

适用范围：

- `备忘`
- `逾期`

默认展示：

- 标题
- 必要状态信息

视觉要求：

- 更轻、更窄、更紧凑
- `逾期` 带弱提醒色
- 不与主区卡片抢视觉权重

### 6.3 默认排序与锚点

所有在线列表统一采用以下规则：

- 新任务或新变更任务追加到底部
- 列表默认优先显示底部
- 新增、流转后自动保持底部可见

适用列表：

- `备忘`
- `逾期`
- `Todo`
- `Doing`
- `Done`

这意味着页面默认优先强调“最新进入状态的任务”。

## 7. 详情抽屉

### 7.1 承载方式

桌面端点击任务卡后，右侧打开 **详情抽屉**。

不采用居中弹窗作为桌面端主承载方式。

### 7.2 设计目标

- 保留看板上下文
- 将信息展示、编辑、状态流转集中到单任务上下文
- 避免每张卡片都沦为控制面板

### 7.3 抽屉内容结构

建议分为 4 个区块：

1. 任务头部
   - 标题
   - 当前状态
   - 分类标签
2. 关键信息
   - 创建时间
   - 截止时间
   - 开始时间
   - 完成时间
   - 必要时显示持续时长
3. 编辑区
   - 标题编辑
   - 分类编辑
   - 时间编辑
   - 备注编辑
4. 操作区
   - 状态流转按钮
   - 删除 / 放弃 / 恢复等操作

### 7.4 打开后的行为

- 抽屉从右侧滑出
- 主工作台保持可见
- 当前选中卡片高亮
- 再点其他卡片时，抽屉内容直接切换
- 点击遮罩或关闭按钮可收起

## 8. 状态流转设计

### 8.1 主链路

主链路固定为：

- `Todo -> Doing -> Done`

这是右侧主工作区唯一强调的流程。

### 8.2 辅助链路

左侧辅助栏承接回流和异常处理：

- `备忘 -> Todo`
- `逾期 -> Todo`
- `任何在线状态 -> 垃圾篓`

垃圾篓为回收态，不属于主视觉常驻区域。

### 8.3 操作按钮规则

操作按钮只在详情抽屉中集中出现。

建议映射：

- `Todo`：开始、完成、删除
- `Doing`：完成、放弃、删除
- `Done`：查看、编辑、删除
- `逾期`：激活、删除
- `备忘`：入列、编辑、删除
- `垃圾篓`：恢复、彻底删除

## 9. 垃圾篓入口

### 9.1 视觉形态

垃圾篓改为小图标按钮，不再占一个完整列或完整区块。

建议位置：

- 左侧辅助栏头部
- 或主工作台右上角工具位

### 9.2 交互

点击垃圾篓图标后，打开垃圾篓抽屉或独立面板。

垃圾篓任务不在主画布常驻露出。

### 9.3 设计意图

- 将空间让渡给 `备忘` 和 `逾期`
- 降低低频删除态的视觉权重
- 保持入口可达但不喧宾夺主

## 10. 列内滚动规则

### 10.1 总体原则

页面整体保持稳定高度，内容溢出时采用 **列内滚动**。

### 10.2 适用范围

- 左侧 `备忘`
- 左侧 `逾期`
- 右侧 `Todo`
- 右侧 `Doing`
- 右侧 `Done`

### 10.3 行为细节

- 列头固定在上方
- 列体内部滚动
- 默认滚动锚点在底部
- 新任务追加后自动滚动到底部附近

这样可以避免某一列任务过多把整页拉成长网页。

## 11. 每日自动流转

### 11.1 Todo -> 逾期

每天结束时，留在 `Todo` 的任务自动转入 `逾期`。

### 11.2 Done -> 归档

每天结束时，留在 `Done` 的任务自动转入 `归档`。

### 11.3 目标

新的一天进入工作台时，只保留新的在线流转任务，不被旧完成项堆积污染。

## 12. 归档模块

### 12.1 模块定位

归档是独立页面或独立视图，不与主工作台混排。

定位：

- 工作台：今天在线流转
- 归档：过去完成沉淀

### 12.2 层级结构

归档按时间层级展开：

- 月
- 周
- 天
- 任务

推荐表现：

- 首屏展示月级摘要
- 展开月后显示周分组
- 展开周后显示每天
- 展开天后显示具体任务列表

### 12.3 归档任务能力

归档任务不是只读仓库，支持：

- 查看详情
- 编辑任务信息
- 恢复回在线状态
- 删除到垃圾篓

### 12.4 恢复规则

默认恢复到 `Todo`。

这样语义最稳定：

- 归档表示“已结束历史”
- 恢复表示“重新进入待处理链路”

### 12.5 归档详情承接

归档任务与在线任务统一使用右侧详情抽屉承接，不再新造一套详情容器。

## 13. 数据与状态影响

为支持归档能力，任务模型或归档索引至少需要新增/明确以下信息：

- `archivedAt`
- `archivedDateKey`
- `archiveMonthKey`
- `archiveWeekKey`
- `archiveDayKey`
- `lastCompletedAt`

可以继续沿用现有任务对象并附加归档字段，也可以单独维护归档索引；本稿不锁死实现方式，但要求满足：

1. 能按月 / 周 / 天快速分组
2. 能从归档任务恢复到在线状态
3. 能保持历史完成时间与归档时间可追踪

## 15. Agent Team 协作规范

### 15.1 事实来源

当前项目内未找到独立的 `agent-team-injection-protocol.md` 文件。

本稿采用以下现有事实作为团队协作依据：

- [AGENTS.md](/D:/lucky/Supplies/个人/APP/监督反思/AGENTS.md)
- [.codex/state/BRIEF.md](/D:/lucky/Supplies/个人/APP/监督反思/.codex/state/BRIEF.md)

### 15.2 团队角色

- 主 agent：规划、拆解、汇总、最终实现、统一写 `.codex/state`
- 调研 agent：只读调研，负责外部检索与事实核查
- 前端 agent：负责任务工作台、抽屉、归档页等前端实现与浏览器验证
- 代码审查 agent：独立审查回归风险、测试缺口、UI 退化
- 用户测试 agent：基于真实浏览器回归核心交互与布局体感

### 15.3 多 agent 约束

- 子 agent 默认只读调研，不直接写 `.codex/state`
- 若涉及状态文件写入，必须由主 agent 串行汇总后统一执行
- 并行任务必须保证文件写入不冲突

### 15.4 前端 agent 执行原则

前端 agent 应优先：

1. 复用当前项目已有页面结构和样式 token，而不是另起一套视觉体系
2. 参考 `前端/web.html` 中已验证的主次分区模板
3. 主动使用浏览器类 MCP 能力做真实页面验证
4. 在现有 skill 足够时，优先使用现有 skills，而不是先安装新 skills

### 15.5 当前可用技能判断

当前项目本地 `.codex/skills` 仅包含状态管理相关技能。

全局已可用且与本任务直接相关的能力包括：

- `brainstorming`
- `writing-plans`
- `requesting-code-review`
- `verification-before-completion`
- `subagent-driven-development`
- 浏览器与页面操作相关 MCP 工具

结论：

- **当前已有 skill/MCP 组合足以支撑本轮 Web 前端改造**
- 在进入实现前，不应默认拉取新的前端 skill 或 plugin
- 只有在以下场景才考虑新增：
  - 现有浏览器验证能力不足
  - 现有前端实现模板无法覆盖需求
  - 新 skill/plugin 明显提升交付质量且能在项目内复用

## 16. 验收标准

### 16.1 布局验收

1. Web 端不再出现五列平均平铺的大白板布局
2. 左侧只常驻 `备忘 / 逾期`，垃圾篓降级为图标入口
3. 右侧稳定展示 `Todo / Doing / Done`
4. 空列不再依赖大面积白卡占位

### 16.2 交互验收

1. 点击任务卡可打开右侧详情抽屉
2. 抽屉中可查看、编辑和执行允许动作
3. 不同状态只显示当前允许的动作组合
4. 切换选中任务时抽屉内容可直接切换

### 16.3 滚动与排序验收

1. 所有在线列表采用列内滚动
2. 新任务追加到底部
3. 列表默认优先显示底部
4. 新增或流转后，底部区域仍保持可见

### 16.4 自动流转验收

1. 每日结束后 `Todo` 自动进入 `逾期`
2. 每日结束后 `Done` 自动进入归档
3. 新一天进入工作台时，旧完成项不再滞留主工作区

### 16.5 归档验收

1. 归档可按月 / 周 / 天展开
2. 归档任务支持查看、编辑、恢复和删除
3. 恢复任务默认回到 `Todo`
