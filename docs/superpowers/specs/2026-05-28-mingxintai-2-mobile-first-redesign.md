# 明心台 2.0 手机优先重构蓝图

日期：2026-05-28

## 1. 核心判断

当前版本已经验证了很多能力：任务、专注、日程、复盘、健康、通知、后端同步、VPS 部署、AI 上下文预留。

但产品形态已经偏离原始目的。问题不是功能不够，而是功能进入页面时缺少主次，导致用户每天打开时不是“立刻知道今天怎么过”，而是在一个复杂系统里找入口。

明心台 2.0 的目标不是继续扩张管理系统，而是重建为一个手机优先、低录入负担、可长期日用的个人生活执行与复盘工具。

一句话目标：

> 每天打开后，用户能立刻看到今天要做什么、现在该做什么、已经怎么花了时间，并能很轻地记录生活与反思。

## 2. 产品原则

1. 手机优先：所有核心操作必须在手机单手或双手短操作内完成。
2. 今日优先：第一屏服务“今天”，而不是服务项目管理或统计报表。
3. 低维护：能自动生成的内容不让用户手填，能默认的配置不强迫用户配置。
4. 少分类：分类只服务用户真实决策，不为了系统完整而拆得很细。
5. 后端隐形：同步、提醒、AI、备份在后台工作，不把技术概念暴露给用户。
6. 允许粗糙记录：生活记录不追求全字段完整，先保证顺手。
7. 先 PWA，后 App：先做手机优先 Web/PWA，稳定后再用 Capacitor 打包 Android。

## 3. 推荐技术路线

### 3.1 第一阶段：手机优先 PWA

使用现代前端重新实现，不再继续维护巨型单文件 HTML。

建议栈：

- 前端：React + Vite + Tailwind + PWA
- 后端：保留 FastAPI + SQLite
- 部署：继续使用 VPS + Nginx + HTTPS
- 通知：继续使用 WxPusher 作为第一版推送
- AI：后续通过后端 API 接入，不在前端直连模型

原因：

- 手机浏览器和桌面浏览器都能用。
- 安卓 15 可以直接添加到主屏幕。
- 迭代速度最快。
- 最适合现有 Codex 前端和浏览器测试能力。

### 3.2 第二阶段：Capacitor 打包 Android

当 PWA 稳定后，再使用 Capacitor 包成 APK。

需要准备：

- Node.js
- Android Studio
- Android SDK
- App 名称、图标、包名
- 签名密钥
- 需要真机或模拟器测试

Capacitor 的意义是：保留 Web 开发效率，同时逐步接入 Android 原生能力。

### 3.3 暂不推荐直接 Kotlin 原生重写

Kotlin / Jetpack Compose 可以做得最原生，但当前阶段成本过高。除非后续明确需要 Android 小组件、深度后台任务、系统级通知操作或离线本地数据库，否则不作为第一阶段方案。

## 4. 信息架构

底部主导航最多保留 5 个入口：

```text
今天
项目
复盘
养息
设置
```

### 4.1 今天

这是默认首页，也是最重要的页面。

内容顺序：

```text
今日概览
├── 当前正在专注 / 休息
├── 备忘
├── 今日执行
├── 未来提醒
├── 想法池入口
└── 今日时间线
```

设计要点：

- 第一屏不展示复杂表格。
- 今日执行支持拖拽排序。
- 备忘用于轻量提醒，不强行要求截止日。
- 未来提醒只露出最近几条，避免压迫感。
- 今日时间线服务回顾，不抢任务主视线。

### 4.2 项目

项目独立，不混进今日列表。

项目用于承载长期目标，例如导师项目、接单论文、个人系统开发。

内容结构：

```text
项目列表
└── 项目详情
    ├── 项目说明
    ├── 活跃任务
    ├── 子任务
    ├── 专注统计
    └── 归档
```

原则：

- 项目在首页不抢位置。
- 今日需要执行的项目任务，会进入“今日执行”。
- 项目可以归档，归档后不进入日常干扰流。

### 4.3 复盘

复盘不是学术复盘，也不是强制表格。

自动展示：

- 当天时间线
- 当天完成任务
- 截至当前仍未完成的任务
- 未完成备忘
- 今日专注总量

用户手写：

- 一个支持 Markdown 的自由输入区
- 不强制模板
- 可以以后让 AI 基于当天数据生成引导问题

### 4.4 养息

养息只保留生活状态的轻记录。

默认模块：

- 睡眠
- 起床
- 运动
- 饮食
- 饮水
- 身高
- 体重

原则：

- 单项记录：如体重、身高、起床时间。
- 多项记录：如饮食、饮水、运动。
- 用户自定义习惯时选择“单项”或“多项”。
- 不把健康记录做成医疗系统。

### 4.5 设置

设置只放低频内容：

- 账号与访问密码
- 通知配置
- AI 配置
- 数据备份与导入
- PWA / App 安装说明

## 5. 核心数据模型

第一版只保留以下实体：

```text
User
Project
Task
Subtask
Reminder
FocusSession
RestSession
ScheduleBlock
DailyReview
HealthRecord
AppSetting
```

任务字段建议：

- 标题
- 所属项目，可为空
- 分区：备忘 / 今日执行 / 未来待办 / 想法池 / 逾期 / 已完成
- 排序
- 状态：未开始 / 专注中 / 暂停 / 已完成
- 计划日，可为空
- 截止时间，可为空
- 是否提醒
- 提醒规则
- 子任务列表

关键规则：

- 切换“进行中”不等于开始专注。
- 只有点击“开始专注”才生成专注状态和时间块。
- 无截止时间的事项不进入逾期。
- 备忘默认不进入逾期，除非用户显式设置提醒时间。
- 今日执行只代表今天希望推进，不代表必须有截止时间。

## 6. 手机端关键交互

### 6.1 快速添加

首页底部固定一个添加按钮。

点击后弹出轻量面板：

```text
标题
类型：备忘 / 今日执行 / 未来待办 / 想法
可选：项目
可选：提醒时间
可选：截止时间
```

默认行为：

- 不填时间就是普通事项。
- 选择今天就是今日执行。
- 选择未来计划日就是未来待办。
- 选择无期限就是想法池。

### 6.2 专注

每个任务和子任务都可以单独开始专注。

开始前可选：

- 不设预计时长
- 25 分钟
- 45 分钟
- 60 分钟
- 自定义

超时规则：

- 预计结束前可提醒。
- 超出后推送提醒。
- 超出 1 小时可自动保护性暂停。
- 下次打开时询问真实结束时间。

### 6.3 一键休息

用户点击休息：

1. 输入预计休息时长。
2. 自动暂停正在专注的任务。
3. 时间线显示休息块。
4. 倒计时正常为绿色，超时为红色。
5. 结束休息后询问是否恢复之前专注。

## 7. 同步与后端边界

后端是权威数据源。

前端不再把 localStorage 当主存储，只保留：

- 登录状态
- 极短期 UI 状态
- 离线临时队列

同步原则：

- 用户操作直接写后端。
- 多端通过服务端事件或低频兜底刷新。
- 冲突不作为常态设计，但保留后端保护。
- 手机和电脑看到的是同一份数据。

## 8. AI 规划

AI 不做第一版核心。

预留方向：

- 每晚基于今日数据生成复盘提示。
- 早上基于任务、截止时间、专注习惯生成今日建议。
- 发现任务快到期但未推进时提醒。
- 对长期项目给出拆解建议。

AI 原则：

- AI 只辅助决策，不自动改用户任务。
- 所有 AI 建议需要用户确认后才落地。
- API Key 只存在后端环境变量或安全配置中。

## 9. MVP 范围

第一版重构只做这些：

1. 手机优先首页：备忘、今日执行、未来待办、想法池、已完成折叠。
2. 项目页：项目列表、项目详情、归档。
3. 任务和子任务：创建、排序、完成、专注。
4. 今日时间线：专注、休息、手动时间块。
5. 复盘页：自动摘要 + Markdown 自由输入。
6. 养息页：默认记录项 + 自定义单项/多项习惯。
7. 设置页：登录、通知、后端状态。
8. PWA 安装能力。

不做：

- 复杂统计仪表盘。
- 大量图表。
- 复杂四象限视图。
- 多用户协作。
- 原生 APK。
- AI 自动规划。

## 10. 验收标准

日常使用验收：

- 用户早上打开手机，10 秒内知道今天先做什么。
- 用户添加一个备忘不超过 5 秒。
- 用户能对今日执行拖拽排序。
- 用户开始和结束专注不需要进入复杂详情页。
- 用户晚上复盘时，系统自动列出今天做了什么和时间怎么花了。
- 手机和电脑打开后看到同一份数据。

设计验收：

- 首页不出现横向表格。
- 单屏信息密度适中，不做大面积装饰卡片。
- 所有点击目标适合手机触控。
- 文本不被时间块、按钮、底栏遮挡。
- 颜色只用于区分状态，不做花哨主题。

技术验收：

- 前端不依赖单文件巨型脚本。
- 后端 API 边界清晰。
- PWA 可安装。
- VPS 部署可复用。
- 浏览器 QA 覆盖手机宽度。

## 11. 后续执行顺序

1. 冻结当前 HTML 为 MVP 归档，不再继续叠功能。
2. 新建 `mingxintai-2` 手机优先前端工程。
3. 先做静态交互原型，不接后端。
4. 用户确认手机端主流程。
5. 接入现有后端最小 API。
6. 迁移或导入旧数据。
7. 部署到 VPS 灰度使用。
8. 稳定后评估 Capacitor 打包 Android。

## 12. 最大风险

最大风险不是技术选型，而是再次把简单生活工具做成复杂系统。

所以 2.0 的硬约束是：

> 每增加一个功能，必须回答：它是否让用户今天更顺手？如果只是让系统更完整，暂不加入。
