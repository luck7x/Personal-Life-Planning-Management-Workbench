> 维护规则：后续开发必须优先对照本文件执行。用户新增反馈后，先更新本 ToDo，再进入实现、审查和测试。

- `[x]` 已完成
- `[~]` 进行中

## 当前路线图

1. [x] 在线模式后端权威
   - 已完成同域 API、浏览器不保存后端 token、启动先读后端。
   - 已加入基础冲突检查：后端状态变化时返回冲突，由用户选择读取后端或覆盖后端。

2. [x] 实时推送替代轮询
   - 目标：PC / 手机一端操作后，另一端无感刷新。
   - 候选方案：SSE 优先，WebSocket 作为后续增强。
   - 验收：普通任务变更无需手动刷新；VPS 不再依赖频繁轮询。
   - 已完成：采用 `/api/events` SSE 推送状态更新时间，前端收到事件后读取 `/api/state`。
   - 已保留兜底：不支持 SSE 或跨域自定义 API 时降级为 60 秒低频检查。

3. [ ] 数据实体拆分
   - 目标：从整包 `workspace_state` 逐步拆为实体表。
   - 第一批：项目、任务、子任务、专注、日程、通知配置。
   - 第二批：健康记录、复盘、投稿、心灯驿。

4. [ ] 操作级 API
   - 目标：用“创建任务 / 修改任务 / 开始专注 / 结束专注 / 排入日程”等接口替代整包覆盖。
   - 验收：不同设备修改不同记录时自动合并，不互相覆盖。

5. [ ] 专注计时服务端时间基准
   - 目标：后端记录 `started_at` 和服务器时间，前端只显示派生计时。
   - 验收：PC / 手机专注显示不再长期差秒；开始、暂停、休息、结束状态一致。

6. [ ] 冲突体验优化
   - 当前：整包级冲突提示。
   - 目标：记录级 / 字段级冲突提示。
   - 规则：低概率同字段冲突时执行用户选择；默认多端不同操作自动合并。

7. [ ] 离线兜底机制
   - 目标：断网时暂存操作队列，恢复网络后提交。
   - 验收：用户能看到哪些操作已提交、哪些需要处理冲突。

8. [ ] 通知任务后台化
   - 目标：到期提醒、逾期提醒、专注预计时长提醒由 VPS 后台定时扫描触发。
   - 验收：浏览器关闭时仍能发送 WxPusher 通知。

9. [ ] AI 分析接口预留
   - 目标：为后续 API 接入 AI 分析提供稳定数据读取接口。
   - 首批接口：今日任务、逾期事项、专注统计、复盘文本、项目进度。

10. [ ] 正式部署清理
    - 目标：移除旧同步文案和临时兼容入口，整理部署文档。
    - 输出：环境变量说明、备份恢复流程、VPS 更新流程、常见故障处理。

## 执行顺序

1. 先完成“实时推送替代轮询”。
2. 再做“数据实体拆分”和“操作级 API”。
3. 然后处理“专注计时服务端时间基准”。
4. 最后做通知后台化、AI 接口和部署清理。

## 每轮开发检查

- 开始前：读取本文件，确认当前最高优先级任务。
- 编码前：使用 `mcp-router` 的 `search_context` 检索相关代码上下文。
- 前端改动前：读取 `instruction.md`，并使用 `frontend-design` skill 的约束。
- 提交前：执行静态语法检查、后端测试和必要的浏览器 QA。
- 完成后：更新本文件的状态和新增反馈。
