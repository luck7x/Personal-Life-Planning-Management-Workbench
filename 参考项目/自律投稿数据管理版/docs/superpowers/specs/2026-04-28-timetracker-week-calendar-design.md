# TimeTracker 式周日历视图设计

## 背景

目标应用是单文件 HTML 版的“自律投稿数据管理版”。当前 `calendar-section` 已有“时间块日历”，但实际是按日期展示的当日时间线，不能直观看到一周时间消耗。

用户确认的方向是 B1：以 TimeTracker 的周时间轴为主视图，右侧保留统计和当天明细。该视图的主要用途是追踪真实时间记录，而不是像 Notion Calendar 一样优先安排未来事项。

## 设计目标

1. 将日历页升级为“周视图 + 右侧统计/当天明细”的时间追踪驾驶舱。
2. 时间块按开始和结束时间映射到纵向时间轴，高度代表持续时长。
3. 视觉参考 TimeTracker：浅色毛玻璃、绿紫渐变、低饱和分类色、高密度但清晰的时间块。
4. 逻辑参考半成品 worktree：周起点、周导航、跨天拆分、事件高度定位、重叠排布。
5. 保持目标 HTML 的低风险改造：只改 `calendar-section` 及其相关 CSS/JS，不重构全应用。

## 非目标

1. 第一版不做拖拽排期。
2. 第一版不做完整月视图或日视图。
3. 第一版不引入 React、Vite 或第三方日历库。
4. 第一版不迁移半成品项目的任务看板、归档页和整套应用框架。

## 信息架构

日历页改为三块：

1. 顶部工具条
   - 上一周
   - 本周
   - 下一周
   - 当前周范围
   - 快速补记入口

2. 主周视图
   - 周一到周日 7 列。
   - 纵轴显示一天内的时间。
   - 时间块使用绝对定位显示在对应日期和时间位置。
   - 同一时间段重叠时横向分 lane 展示。

3. 右侧面板
   - 今日计时器状态。
   - 当天明细列表。
   - 本周分类统计。
   - 投稿 deadline 轻提醒。

## 数据模型

第一版不新增顶层 state，优先复用现有数据：

1. `state.timeBlocks`
   - 主数据源。
   - 每个记录包含 `date`、`start`、`end`、`category`、`title`、`notes`、`minutes`。

2. `state.focus.active`
   - 表示当前正在计时的任务。
   - 右侧计时器读取它显示实时计时。

3. `state.focus.sessions`
   - 作为日历辅助数据源。
   - 历史专注记录可叠加到周视图。

4. `state.tasks`
   - 需要从 `todo | done` 扩展为支持 `doing`。
   - 当任务从 `todo` 流转到 `doing` 时，自动写入 `state.focus.active` 并开始计时。
   - 当 `doing` 流转到 `done` 时，结束计时并写入 `state.focus.sessions`。

5. `state.submissions`
   - 仅作为右侧提醒或日历角标。
   - 不作为主时间块，避免把 deadline 和真实时间消耗混在一起。

## 计时器交互

任务状态流转：

```text
todo -> doing -> done
```

规则：

1. `todo -> doing`
   - 自动开始计时。
   - 写入 `state.focus.active`。
   - active 内容至少包含任务标题、任务 id、开始时间戳、日期。

2. `doing -> done`
   - 自动结束计时。
   - 根据开始和结束时间计算分钟数。
   - 写入 `state.focus.sessions`。
   - 同步刷新周视图、右侧计时器和任务列表。

3. `doing -> todo`
   - 视为暂停或撤回。
   - 第一版建议弹出确认，避免误丢正在计时的数据。
   - 若确认撤回，则清空当前 active，不写入 sessions。

4. 同时只能有一个 active 任务。
   - 如果已有任务正在 doing，再启动另一个任务，需要先结束或放弃当前任务。

## 视觉方向

日历页可以保留全应用整体背景，但内部切换为独立视觉语言：

1. 主日历容器
   - 半透明白色背景。
   - 玻璃描边。
   - 柔和阴影。
   - 圆角控制在 18 到 24px。

2. 时间块
   - 分类色低透明填充。
   - 细白色边框。
   - 标题加粗，时间信息次级显示。
   - 短事件使用紧凑行内显示。

3. 时间网格
   - 轻量横线和竖线。
   - 周末使用轻微红灰色文本。
   - 当前日用白色描边或淡背景强调。

4. 右侧面板
   - Timer、Stats、当天明细分为三个玻璃面板。
   - Timer 使用圆环进度视觉，但不强求精确进度含义。

## 测试策略

由于目标是单文件 HTML，第一版测试采用轻量验证：

1. 抽取纯函数测试
   - 周一作为周起点。
   - 上一周/下一周日期正确。
   - 时间块位置和高度计算正确。
   - 重叠事件 lane 分配正确。

2. 浏览器手动验证
   - 添加时间块后出现在正确日期和时间。
   - 周导航能切换数据。
   - 任务从 todo 到 doing 后计时器开始。
   - doing 到 done 后生成专注记录。
   - 右侧统计随数据变化刷新。

3. 回归检查
   - 起居考勤月历不受影响。
   - 投稿管理不受影响。
   - 数据导入导出仍能包含新增或扩展字段。

## 已确认决策

1. 选择 B1，不选择月历优先。
2. 第一版主打时间追踪，不主打排期。
3. 计时器由任务状态 `todo -> doing` 自动触发。
4. 半成品 worktree 只迁移逻辑，不迁移 UI。
5. TimeTracker 只借鉴视觉，不迁移 PySide 实现。
